# 扩展指南

## 开发基线

当前扩展应以以下事实为准：

- UI 主控制器当前在 `plugin.gd` 的内联 `DockController`
- `ui/orbit_dock.tscn` 当前为纯 UI 场景
- 重构计划将 `DockController` 提取到 `ui/dock_controller.gd`，并在 `ui/pages/` 下建立 5 个页面控制器

如果扩展 UI，先参考 `agent_workflow/10_开发计划/03_当前版本更新实施手册.md` 中的重构批次安排，避免与重构中的文件产生冲突。

## 新增本地工具

1. 在 `core/tools/local_tools.gd` 增加 `get_definitions()` 项
2. 在 `execute()` 增加 action 分发
3. 若涉及写入，必须走 `ChangeSetService`
4. 在 `docs/模块/本地工具与上下文索引.md` 更新说明
5. 更新 `docs/测试/测试策略.md` 中对应测试项

## 新增远端服务器

1. 在设置中增加 `remote_servers.{id}`
2. 确认 endpoint 填写的是根地址，而不是 `/mcp`
3. 运行“刷新远端工具”
4. 确认工具目录中出现 `remote:{id}:*`
5. 更新 `docs/模块/远端MCP互通.md`

## 新增 Provider

1. 新建 `core/provider/{xxx}_provider.gd` 继承 `ProviderAdapter`
2. 实现 `get_id()` 与 `send_chat()`，必要时实现 `send_chat_stream()`
3. 在 `plugin.gd` 注册实例到 `_execution_service.setup(...)`
4. 在默认设置中补齐 provider 配置
5. 在 UI 的 provider 选择器中展示
6. 更新：
   - `docs/模块/Provider层.md`
   - `docs/测试/测试策略.md`

## 新增 OpenAI Compatible 预设 Provider

例如 DeepSeek 这一类，只需：

1. 在 `core/types.gd` 添加 provider 配置
2. 在 `plugin.gd` 的 provider map 中映射到 `_openai_provider`
3. 补充默认 `base_url`、`model`、`api_key_env`
4. 更新 `Provider层.md`

这种路径适合：

- 协议兼容 OpenAI
- 只需差异化默认配置
- 不需要独立的响应解析逻辑

## 扩展 Sessions 页面

当前 Sessions 页包含：

- Active 列表
- Archived 列表
- New Session
- Open Session
- Archive Selected
- Delete Archived
- Rename

若继续扩展：

1. 先补 `_settings["sessions"]` 数据字段
2. 再补 `DockController.apply_model()` 渲染逻辑
3. 再补对应 `_on_session_*` 主控处理函数
4. 最后补文档与测试

## 扩展导航页

新增页面时至少同步修改：

- `ui/orbit_dock.tscn` 页面节点结构
- `DockController.PAGE_*` 常量
- `_assign_nodes()`
- `_bind_events()`
- `_set_page()`
- `_apply_nav_state()`

## 常见错误与修复记录

以下记录来自本轮 `Batch 4` 到 `Batch 6` 重构过程，后续若再次调整启动链、Provider 或 Dock 场景拆分，优先先对照这里排查。

### 1. `plugin.gd` 启动期脚本不可实例化

典型报错：

- `OpenAIProvider cannot instantiate`
- `AnthropicProvider cannot instantiate`
- `Service initialization failed. Check script parse errors.`

实际原因：

- 在当前 Godot 运行环境下，`can_instantiate()` 与部分 `GDScript.new()` 组合并不稳定。
- 即使 `script_inspect` 能解析脚本，也不代表启动期通过 `Script` 变量动态实例化一定成功。

当前稳定修复：

- `plugin.gd` 中不要对服务脚本走复杂的动态工厂链。
- 优先保持入口脚本简单，避免在 `_init_services()` 中叠加多层 `load(...) -> Script -> new()` 包装。
- 若重构后再次出现同类报错，先回退到“入口直接装配、减少中间层”的稳定结构，再逐步拆分。

### 2. `plugin_runtime.gd` 导致插件入口解析失败

典型报错：

- `Could not resolve script "res://addons/orbit_pilot/core/plugin/plugin_runtime.gd"`
- `Invalid call. Nonexistent function 'save_settings' in base 'Nil'`

实际原因：

- `plugin.gd` 在解析阶段通过 `preload(...)` 强依赖 `plugin_runtime.gd`。
- 一旦 `plugin_runtime.gd` 资源状态、缓存或实例化链异常，整个插件入口会被连带拖死。

当前稳定修复：

- `plugin.gd` 不再依赖 `plugin_runtime.gd` 启动。
- Settings 保存、日志、远端 MCP 刷新、启动校验暂时收回 `plugin.gd` 本体维护。
- 后续若再次尝试抽出 `plugin_runtime.gd`，必须先满足：
  - 入口脚本在缺失该模块时仍可启动。
  - `_exit_tree()` 对空运行时对象有防御。
  - 不允许在解析阶段形成新的强耦合单点。

### 3. `provider_base.gd` 直接继承链导致 Provider 本身不可实例化

典型报错：

- `OpenAIProvider cannot instantiate`
- `AnthropicProvider cannot instantiate`
- `Invalid call. Nonexistent function 'new' in base 'GDScript'`

实际原因：

- 将 `openai_provider.gd` / `anthropic_provider.gd` 直接改成 `extends "provider_base.gd"` 后，在当前环境中出现了脚本本身不可实例化的问题。
- 同时，对 `provider_base.gd` 继续走 `load(...)` 或 `Script.new()` 也会放大该问题。

当前稳定修复：

- Provider 当前仍使用 `provider_adapter.gd` 作为稳定适配层。
- `provider_base.gd` 可以保留为重构试验产物，但不要再接回主启动链，除非先完成独立验证。
- 新增 Provider 时，先沿用 `ProviderAdapter` 路径，不要直接继承 `provider_base.gd`。

### 4. Settings 丢失或 `_save_settings()` 收到 `Nil`

典型报错：

- `Cannot convert argument 1 from Nil to Dictionary`

实际原因：

- 插件在启停、重载或初始化异常边界上，`_settings` / `_logs` 可能尚未恢复，但 `_save_settings()` / `_push_log()` 已提前触发。

当前稳定修复：

- `_save_settings()` 必须先确认 `_settings is Dictionary`，必要时重新加载默认设置。
- `_push_log()` 必须对 `_logs` 和 `_settings` 做兜底初始化。
- 任何新抽出的会话、运行时、设置模块，都不能假设插件主状态一定已初始化完成。

### 5. Provider 联通时报 `http_request_failed`

典型现象：

- UI 配置看起来正确，`MCP main: ok`
- 发送消息后日志出现 `run_status:provider_request_started`
- 随后出现 `agent_error:http_request_failed`

排查结论：

- 这通常不是用户把 `model` / `base_url` / `remote endpoint` 填错了。
- 更常见的是 Provider 请求链、适配层、`HTTPRequest` 发起方式或重构后引用关系异常。

排查顺序：

1. 先确认不是 `missing_api_key`。
2. 再确认不是 `http_error_status`（401/403/404）。
3. 若是 `http_request_failed`，优先检查 Provider 实现与适配层，不要先要求用户反复改表单。
4. 若当前预设为 `deepseek`，确认 `base_url` 使用 `https://api.deepseek.com/v1`，不要省略 `/v1`。

## 文档同步要求

任何涉及以下内容的改动，都必须同步更新文档：

- 运行时入口变更
- Provider 接入方式变更
- MCP endpoint 约定变更
- 会话数据结构变更
- ChangeSet 审批链变更
- UI 页面结构变更
