# 事件模型

## 目标

执行层与插件主控之间通过统一事件对象通信，避免 Provider、ToolRouter、UI 三者直接耦合。

## 事件对象结构

```json
{
  "run_id": "run_...",
  "kind": "message_delta",
  "payload": {}
}
```

字段：

- `run_id`：一次回合唯一标识
- `kind`：事件类型
- `payload`：事件负载

## 当前事件类型

### `status`

示例：

```json
{ "message": "provider_request_started" }
```

用途：

- 给日志和状态栏提供粗粒度阶段信息

### `message_delta`

示例：

```json
{ "delta": "partial text" }
```

用途：

- 用于流式预览文本累加
- 当前会进入 `_stream_preview`

### `tool_call`

示例：

```json
{ "tool": "remote:main:scene_management", "args": {} }
```

用途：

- 记录工具调用开始
- 便于调试模型返回的工具决策

### `tool_call_delta`

示例：

```json
{ "tool": "remote:main:script_edit_gd", "arguments_delta": "..." }
```

用途：

- 记录流式工具参数增量

### `tool_result`

示例：

```json
{ "tool": "local:search_files", "result": { "success": true } }
```

用途：

- 输出工具结果
- 写入日志或工具结果面板

## 当前消费规则

- 以 `run_id` 区分不同回合事件
- `message_delta` 只进入当前 active run 的预览缓冲
- `tool_call` / `tool_call_delta` / `tool_result` 当前主要写日志
- 回合结束、失败、取消时必须清空 active run 状态与 preview

## 调试建议

当出现“回复重复显示”时，先检查：

1. `message_delta` 是否已累加到 `_stream_preview`
2. 回合结束后是否清空 `_stream_preview`
3. UI 是否同时渲染了正式 assistant message 和 preview
