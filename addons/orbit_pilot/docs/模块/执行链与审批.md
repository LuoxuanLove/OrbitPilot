# 执行链与审批

## 执行策略

文件：`core/execution/execution_service.gd`

当前支持：

- `analyze`：仅模型分析，不执行写入。
- `propose`：允许工具调用，但拦截 mutating 写入工具。
- `apply_with_approval`：允许完整链路，但本地写入仍需审批。

## 当前主回合流程

1. 聊天页输入 prompt。
2. `plugin.gd` 获取当前 active session。
3. 追加 user message 到 `session.messages`。
4. 组装 Provider 配置与统一工具目录。
5. 调用 `ExecutionService.execute_turn(...)`。
6. 执行层：
   - 请求 Provider
   - 如有 tool call，则转发给 `ToolRouter`
   - 收集 tool_result
7. 若涉及写入，则由本地工具创建 pending change set。
8. 主控刷新 Review 页面与日志。
9. 审批通过后写入磁盘。

## 流式预览

主控当前通过 `_stream_preview` 存储未完成的 assistant 增量文本。

约束：

- `message_delta` 到来时，追加到 `_stream_preview`
- 回合成功结束后，必须清空 `_stream_preview`
- 回合失败或取消后，也必须清空 `_stream_preview`

原因：

- 如果不清空，UI 会同时渲染正式 assistant message 和残留 preview，看起来像 assistant 回复重复出现两次

## 事件回调

当前执行层会回调：

- `status`
- `message_delta`
- `tool_call`
- `tool_call_delta`
- `tool_result`

主控当前处理方式：

- `status` -> 写日志
- `message_delta` -> 更新 `_stream_preview`
- `tool_call` / `tool_call_delta` / `tool_result` -> 写日志

## ChangeSet 审批

文件：`core/changes/change_set_service.gd`

当前流程：

1. 本地写入类工具创建 `ChangeSet`
2. 状态为 `pending`
3. Review 页展示 pending 列表与 diff 预览
4. 用户审批：
   - `Approve` -> 应用并落盘
   - `Reject` -> 标记拒绝，不落盘
5. 支持最近一次应用内容回滚

## 当前 UI 行为

- 聊天页不直接写入文件。
- Review 页承担变更审批入口。
- Tools 页可以手动执行工具，但结果仍通过结构化 result 回显。

## 取消机制

当前状态字段：

- `_active_run_id`
- `_is_run_active`

流程：

1. 发送 prompt 时创建 `run_id`
2. UI 可点 `Cancel`
3. 主控调用 `ExecutionService.cancel_run(run_id)`
4. 执行层在关键阶段检查取消标记
5. 取消后收尾并刷新 Dock 状态

## 失败处理

### Provider 请求失败

当前行为：

- 格式化错误信息
- 在当前会话追加一条 assistant 消息：`Request failed: ...`
- 写日志
- 清空 active run 状态

### Tool 执行失败

当前行为：

- 工具结果保留错误结构
- 日志记录失败信息
- 插件继续存活

### 审批失败

当前行为：

- `ChangeSetService` 返回失败结构
- UI 刷新当前 review 状态
- 不影响其他 session 或插件运行
